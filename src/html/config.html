Basic Configurations

Api Keys

First step is to create an account here and create your first application , so you will get your APP Key & APP Secret.

Note about keys

Your App Secret should be strictly confidential, don't share it with any one or posting it public at any way.

Action / Response

Shieldfy lets you control the action should the package do when spotting dangerous activities

Available Actions: ( Block | Listen ) the default action is block will block the dangerous activities , the other action listen will record the threat and notify you but will not block the request and will let the user continue to your application. 

<?php 
use Shieldfy\Guard;
Guard::init([
        'app_key'=>'APP_KEY',
        'app_secret'=>'APP_SECRET',
        'action'=>'listen' //default is block
])->catchCallbacks();

Headers Control

For security reasons Shieldfy expose number of headers to enforce some security practices.

The headers exposed by default

  • X-XSS-Protection
  • X-Content-Type-Options
  • X-Frame-Options

If for some reason you don't want this headers to be exposed or you want to expose another value for it, you can control it. Just pass array for the headers and pass the changed value or false if you want to disable it completely. 

<?php 
use Shieldfy\Guard;
Guard::init([
        'app_key'=>'APP_KEY',
        'app_secret'=>'APP_SECRET',
        'headers'=>[
            'X-Frame-Options' => 'DENY', //this header value will be DENY
            'X-XSS-Protection' => false //this header will not be shown
        ]
])->catchCallbacks();

Using Cache

Shieldfy SDK rely on cache for tracking users footprints locally on your server , by default shieldfy uses files cache in the sdk folder /path/to/sdk/tmp. But you can change the cache driver or use your own custom cache

File Cache

You can use file cache easily be setting up Shieldfy Cache manager

<?php 
use Shieldfy\Guard;
use Shieldfy\Cache\CacheManager;

$cache = new CacheManager();
$cache = $cache->setDriver('file', [
    'path'=> realpath('path/to/cache/directory').'/', //make sure this dir is writable
]);

$guard = Shieldfy\Guard::init([
    'app_key'=>'APP_KEY',
    'app_secret'=>'APP_SECRET',
],$cache)->catchCallbacks();

Memcached

If you are using Memcached , you can configure shieldfy to work with Memcached

<?php 
use Shieldfy\Guard;
use Shieldfy\Cache\CacheManager;

$cache = new CacheManager();
$cache = $cache->setDriver('memcached', [
    'servers'=> [

    ]
]);

$guard = Shieldfy\Guard::init([
    'app_key'=>'APP_KEY',
    'app_secret'=>'APP_SECRET',
],$cache)->catchCallbacks();

Build your own cache

If you want to use your own custom cache for shieldfy for example using redis or custom mongodb collection .. etc , you can extend the driver from Cache manager

First you need to create a class implements CacheInterface

<?php 
namespace MyOwnApp\Cache;
use Shieldfy\Cache\CacheInterface;

class MyOwnCustomCache implements CacheInterface
{
    private $adapter = null;
    private $timeout = 3600;
    public function __construct($config = [], $timeout = '')
    {
        if ($timeout) {
            $this->timeout = $timeout;
        }
        //.... your initial configurations
    }

    public function has($key)
    {
        //return true if the key exists
    }
    public function set($key, $value)
    {
        //sets key with the givin value
    }
    public function get($key)
    {
        //get the key value
    }
}

Then setup the Cachemanager to use your MyOwnCustomCache 

<?php 
use Shieldfy\Guard;
use Shieldfy\Cache\CacheManager;

$cache = new CacheManager();
$cache = $cache->extendDriver(\MyOwnApp\Cache\MyOwnCustomCacheMyOwnCustomCache::class, $yourconfigurations);

$guard = Shieldfy\Guard::init([
    'app_key'=>'APP_KEY',
    'app_secret'=>'APP_SECRET',
],$cache)->catchCallbacks();

Callbacks

Shieldfy communitcate back to the SDK in three different callbacks Health check & Logs grabbing & Update for rules

All Callback communication is encrypted and done over ssl if available.

Health Check Callback

Health check callback is just a ping to your website , the SDK catch the request and response with the status : ok and the version of the SDK

When we got the right response we know its all ok , but if we don't we know something not right. so we can address the issue quickly with you.

Logs Callback

Shieldfy SDK produce number of log files when something went wrong or when needs to store some SDK related info, and don't interrupt the user expirience in any way .

So this logs is an indicator of how the sdk works so we can improve it

Logs Privacy

All logs will be a package related logs , mostly errors and notices for the package preformance , it will not include any users info or other info consider private.

Update Callback

This is a very important callback although shieldfy rely on so many metrics to identify attacks , Shieldfy still uses some rules at the heart of attack identification process .

The callback will be triggered if we release a new update in the application/attacks rules , so your application be up to date with the security automatically.
